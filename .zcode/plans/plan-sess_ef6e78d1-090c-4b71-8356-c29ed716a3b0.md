## 目标
封装一个干净、易用的「窗口预定位」能力：侧边窗口在**首次显示前**就被放到目标位置（主窗口右侧），彻底消除「先在默认位置闪一下、再跳到右边」的视觉跳动。封装要好用到调用方只需一行注册，不暴露任何 Win32 细节。

## 根因回顾（已验证）
Gio 在 Windows 上创建窗口的时序（`gioui.org@v0.10.1/app/os_windows.go` `newWindow`）：
```
CreateWindowEx(…, CW_USEDEFAULT, …)   // 创建，不可见（style 无 WS_VISIBLE）
Configure(options)                    // SetWindowPos(默认位置) + ShowWindow(SW_SHOWNORMAL) ← 在默认位置显示！
Win32ViewEvent                        ← 我们最早拿到 HWND 的时机，但窗口已显示
```
Gio 没有 Position option，`Win32ViewEvent` 晚于 `ShowWindow`，所以当前在 ViewEvent 里 `MoveWindow` 必然迟到一帧 → 闪一下。

## 技术方案：WinEvent 钩子 + 窗口子类化

在 `CreateWindowEx` 和 `ShowWindow` 之间介入：`SetWinEventHook(EVENT_OBJECT_CREATE, WINEVENT_INCONTEXT)` 的回调由触发线程（Gio 窗口线程）**同步**调用，而 `EVENT_OBJECT_CREATE` 在 `CreateWindowEx` 期间发出、早于 `Configure`/`ShowWindow`。因此在回调里立即 `SetWindowSubclass` 子类化新窗口，子类过程拦截 `WM_SHOWWINDOW`，在首次显示前 `SetWindowPos` 到目标位置，再放行显示。

**怎么识别"这是侧边窗口"**：用 `GetWindowThreadProcessId` 确认是本进程窗口 + `GetWindow(hwnd, GW_OWNER)` 判断它没有 owner（顶层），再配合「该窗口是新建的、且不是主窗口 HWND」即可命中。更稳妥的做法：**主窗口先创建并已知 HWND**，钩子回调里凡新建的、`GW_OWNER==0`、`!= 主窗口HWND` 的顶层窗口都视为待定位窗口，套用一个注册的「定位函数」。

## 封装 API 设计（platform 层，跨平台）

### `desktop/internal/platform/platform_windows.go`（+ 其他平台的 no-op）
新增一个独立、自包含的定位器：

```go
// PositionPreset 是窗口「首次显示前」的预设定位策略。
type PositionPreset int

const (
    // PositionRightOfOwner 把窗口紧贴在 owner 窗口右侧，与 owner 等高。
    PositionRightOfOwner PositionPreset = iota
)

// WindowPositor 在窗口首次显示前把它定位到目标位置，避免「先在默认位置闪现再跳」。
// 它通过一个进程级 WinEvent 钩子在窗口创建瞬间拦截并定位，对调用方完全透明。
type WindowPositor struct{ /* 内部状态，不导出 */ }

// NewWindowPositor 创建并安装钩子。在应用启动、主窗口 HWND 已知后调用一次。
// mainHandle 是主窗口的 HWND，用作定位参照和"排除自身"的依据。
func NewWindowPositor(mainHandle Handle) *WindowPositor

// ExpectRightOfOwner 声明：接下来即将创建的子窗口应贴在 owner（主窗口）右侧。
// 在创建 *app.Window 之前调用。返回一个 cancel 函数，窗口定位完成后或不再需要时调用，清理子类化。
// 这是最易用的形式：调用方只需「我要开一个右侧窗口」+ 一个 owner。
func (p *WindowPositor) ExpectRightOfOwner(owner Handle) (cancel func())
```

调用方（`side_window.go`）用法极简：
```go
// 在 go sw.run() 之前：
cancel := sw.app.Positor.ExpectRightOfOwner(sw.app.OwnerHandle)
// ... run() 创建窗口，钩子在首次 ShowWindow 前定位到 owner 右侧 ...
// 定位完成（ViewEvent 后或窗口关闭时）cancel()
```

### `desktop/internal/platform/windows/position.go`（新增，Windows 实现）
- syscall 绑定：`SetWinEventHook`、`UnhookWinEvent`、`SetWindowSubclass`、`RemoveWindowSubclass`、`DefSubclassProc`、`GetWindowThreadProcessId`、`GetWindow`。
- 常量：`EVENT_OBJECT_CREATE`、`WINEVENT_INCONTEXT`、`WM_SHOWWINDOW`、`GW_OWNER`、`SWP_NOZORDER|SWP_NOACTIVATE`。
- 钩子线程：复用项目既有模式——专用 goroutine + `runtime.LockOSThread()`（钩子回调对线程有亲和性）。
- 子类过程：`syscall.NewCallback`，拦截首个 `WM_SHOWWINDOW`（`lParam==0` 即由 ShowWindow 触发），调注册的定位函数（算出 owner 右侧坐标 → SetWindowPos），随后 `DefSubclassProc` 放行，并 `RemoveWindowSubclass` 自卸载（只拦首次）。

### 其他平台
- `platform_other.go` / 对应文件提供 no-op `WindowPositor`（`ExpectRightOfOwner` 返回 no-op cancel），保证非 Windows 构建通过、API 一致。

## 行为变化
- 侧边窗口首次打开：**无闪烁**，直接在主窗口右侧出现。
- 主窗口自身不受影响（钩子用 owner/HWND 排除）。
- 对话框不使用此机制（它们本就居中，无此问题），但 API 通用，将来可复用。

## 边界与清理
- `cancel()` 必须 idempotent 且安全（窗口已关闭、重复调用都 OK）。
- 钩子卸载：进程退出自然回收；侧边窗口关闭时 `cancel` 解除子类化，避免泄漏。
- 只拦**首次** `WM_SHOWWINDOW`，之后窗口正常工作（拖拽、重绘不受影响）。
- 定位坐标在钩子回调里实时算（owner 当前几何），不依赖创建前的快照——用户拖动主窗口后再开侧边窗仍正确。

## 验证
- `GOOS=windows go build ./...` + `go vet` + `gofmt` 全过。
- 非平台构建（无 build tag 文件）不破。
- 编译复制 `todo-desktop22.exe`，实测：首次开侧边窗无闪烁、位置正确；多次开关仍正确；主窗口移动后再开侧边窗位置跟随。

## 不做的事
- 不动主窗口定位、不动 dock/拖拽、不动业务逻辑。
- 不改 Gio 内部（只在本进程用户态加钩子，不 patch 依赖）。
